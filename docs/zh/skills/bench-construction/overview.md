# Benchmark 数据构建流程

## 为什么重要

构建流程是 Benchmark 论文的技术贡献，相当于技术论文中的方法章节。审稿人对这一
章节的审查最为严苛，因为流程描述含糊、无法复现或质量控制薄弱，就无法支撑后续发现。
建立在审稿人无法信任的流程之上的 Benchmark，其发现审稿人也不会引用。NeurIPS
Datasets and Benchmarks、ICLR 和 CVPR 上的常见失败模式包括：两段话的流程描述
缺少流程图，标注者间一致性数字缺失或缺乏辩护，以及合成数据生成没有对抗验证来
核实合成数据是否真实地模拟了真实数据。

这个技能将来自
[bench-design](../bench-design/overview.md) 的设计文档转化为完整的流程规格：
构建范式、每阶段的输入/操作/输出及明确质量门控、文档化的质量控制策略、论文第 2.3
节的统计特征规划，以及审稿人能在一分钟内读懂的流程图（Figure 2）。这个技能顶部
的强制门禁拒绝在没有完整设计文档的情况下开始流水线工作，因为流水线以设计的分类学
和评估框架为构建目标。

## 核心框架

技能的结构镜像了构建流程的审稿人检查清单：选择范式、规格化各阶段、记录质量控制
策略、规划统计特征描述、设计流程图。

### 流水线设计原则

每条构建流水线都必须满足三个属性：

1. **可复现性**：另一个团队可以遵循相同步骤得到可比较的数据。所有提示、脚本和
   标注指南公开发布。
2. **可扩展性**：流水线可以在不成比例增加人工工作量的情况下生成更多数据。瓶颈要明确说明。
3. **质量保证**：每个数据点都经过明确的质量门控。没有通过门控的数据不能进入 Benchmark。

### 三种已验证的构建范式

三种范式覆盖了常见的 Benchmark 形态，每种附一篇参考论文和适用场景。

**范式一：逆向合成（StatQA 方法）。** 先固定答案，再生成问题。

```
定义答案空间 -> 生成匹配条件 ->
组合问题         验证唯一性
（枚举）          （场景）
（自然语言包装）   （唯一正确答案）
```

适用场景：答案空间可枚举（统计方法、图表类型、代码模式），且需要精确控制难度
分布和类别平衡。核心优势是构造上即具有无歧义的基准真值：答案在问题存在之前就
已知晓，所以"正确"的含义没有歧义。

**范式二：可控注入（nvBench 2.0 方法）。** 从干净种子出发，系统性注入目标现象。

```
收集干净种子 -> 定义注入类型 ->
以校准级别注入    验证自然度
（无歧义）         （例：4 种类型）
（例：5 个严重级别）（人工检查）
```

适用场景：研究某种特定现象（歧义、噪声、偏差、对抗扰动），需要跨该现象的可控
严重程度梯度。核心优势是变量隔离：目标现象可以独立于混杂因素单独研究，因为
干净种子充当对照条件。

**范式三：自适应生成加专家标注（VisJudge-Bench 方法）。** LLM 生成候选数据，
人工在多个阶段进行标注。

```
LLM 生成候选 -> 第一阶段筛选 ->
第二阶段细粒度评分   第三阶段交叉验证
（适应内容）          （专家过滤）
（多维度）            （分歧解决）
```

适用场景：任务需要无法完全自动化的细致人类判断，质量依赖于跨多个主观维度的
专家评估。核心优势是结合 LLM 的规模效率与人工判断的质量：两者单独都无法为
主观任务产出审稿人信任的数据。

技能要求用户选择一种范式（或说明混合多种要素的理由），并描述数据来源和流程草图。

### 流水线各阶段规格

无论选择哪种范式，每个阶段都用这个模板记录：

| 阶段 | 输入 | 操作 | 输出 | 质量门控 |
|---|---|---|---|---|
| **1. 来源选择** | 原始数据来源 | 过滤、许可证检查、去重 | 干净的来源语料库 | 覆盖度审计、许可证检查 |
| **2. 种子生成** | 来源语料库 | [范式特定方法] | 初始样本 | 格式验证、唯一性 |
| **3. 富化** | 初始样本 | 添加元数据、标签、标记 | 带标注样本 | 分类学覆盖平衡 |
| **4. 质量控制** | 带标注样本 | 人工审核、自动验证 | 验证后样本 | IAA 达到或超过阈值 |
| **5. 数据划分** | 验证后样本 | 分层训练/验证/测试划分 | 最终数据集 | 分布平衡检查 |

每个阶段，技能推动用户说明由谁执行（自动脚本、LLM、人工标注者、领域专家），
每个样本和总体需要多长时间，可能出什么问题以及如何处理失败。流程描述含糊是
最常见的拒稿理由之一。

### 质量控制策略

至少记录三种策略。审稿人对此审查非常严格，因为质量控制薄弱意味着发现不可信。

| 策略 | 方法 | 适用场景 |
|---|---|---|
| 标注者间一致性 | 多位标注者对同一样本标注；报告 Cohen's kappa、Fleiss' kappa 或 ICC | 任何人工标注任务 |
| 专家抽检 | 领域专家审核随机抽取的 10% 或以上子集 | 半自动化流水线 |
| 对抗验证 | 测试分类器是否能区分合成数据和真实数据 | 合成数据生成 |
| 自动完整性检查 | 基于规则的验证：格式、范围、一致性、去重 | 所有流水线 |
| 小规模试点 | 全量构建前的小规模试运行（50-100 个样本） | 复杂多阶段流水线 |
| LLM 交叉验证 | 使用不同 LLM 家族验证 LLM 生成的数据 | 含 LLM 的流水线 |

标注者间一致性是任何涉及人工标注的 Benchmark 的最低要求。专家抽检对半自动化
流水线强烈推荐。只要使用了合成数据，就必须进行对抗验证，因为审稿人会追问
合成数据是否真实模拟了真实数据。

### 统计特征描述（第 2.3 节内容）

构建完成后，呈现全面的统计数据。以下"必须有"列表对应最终论文的 Table 2 和
Figures 3-4：

- 按数据划分（训练/验证/测试）的样本数量。
- 跨分类学类别的分布（柱状图或饼图）。
- 难度分布（直方图）。
- 文本长度或复杂度分布。
- 与现有 Benchmark 的规模对比。

推荐补充项：词云或主题分布、来源多样性指标、2-3 个有代表性数据示例（作为图）、
标注时间和成本统计（体现透明度）。

### 流程图（Figure 2）

流水线必须可视化。设计指南：

- 左到右或上到下的流动，方向清晰。
- 每个步骤是带简短描述的标注方框。
- 步骤间的箭头标注数据量（例如："10K -> 过滤 -> 8.5K"）。
- 质量门控在视觉上有所区分（菱形、彩色边框或门禁图标）。
- 一个具体示例展示一个样本如何流过整条流水线。
- 范式标签（逆向合成、可控注入或自适应生成）出现在图的顶部。

## 核心原则

- **三种已验证范式**：逆向合成、可控注入、自适应生成加专家标注。选择一种与
  任务形态匹配的，或说明组合的理由。
- **五阶段模板**：来源选择、种子生成、富化、质量控制、数据划分。每个阶段都
  有明确的输入、操作、输出和质量门控。
- **至少三种质量控制策略**：标注者间一致性加两种针对流水线风险点的策略。
- **流程图为必须项**：Figure 2，箭头上标注数据量，质量门控在视觉上有所区分。
  审稿人需要能在一分钟内读懂流水线。
- **数据量标注**：每个箭头显示流经该阶段的数据量，让读者能追踪保留率。

## 示例

考虑 nvBench 2.0 团队完成设计后的情况。盲区模式是假设违背：现有 Text-to-Visualisation
Benchmark 假设单一正确输出，但真实查询具有歧义性。设计采用 Pattern 2（现象类型 ×
严重程度谱系），四种歧义类型乘以五个严重程度级别。

**范式选择。** 可控注入完全匹配。团队从 nvBench 1.0 的无歧义查询-图表对语料库出发，
定义四种歧义注入类型（图表类型歧义、属性歧义、聚合歧义、过滤歧义），并以五个
校准严重程度级别进行注入。干净种子充当对照条件。

**流水线各阶段。** 第 1 阶段（来源选择）：从 nvBench 1.0 中取 7,247 个无歧义对，
过滤质量。第 2 阶段（种子生成）：对每个干净种子，使用基于规则加 LLM 的混合流水线，
为每种注入类型和严重程度级别生成一个歧义变体。第 3 阶段（富化）：对每个歧义变体，
标注有效图表解释集合，附偏好排名。第 4 阶段（质量控制）：三位标注者对每个歧义
变体在自然度（读起来是否像真实用户查询？）和有效性（标注的解释是否都合理？）
上进行评分，报告 Fleiss' kappa。第 5 阶段（数据划分）：按现象类型和严重程度级别
分层划分。

**质量控制。** 记录了三种策略：（1）三位标注者在自然度和有效性上的 Fleiss' kappa，
最低阈值 0.6；（2）可视化博士生抽检 10%；（3）使用不同 LLM 家族重新生成图表
备选项，确认与标注集的重叠度。

**统计特征描述。** 总规模（每种注入类型和严重程度级别的歧义变体数量），二十个评估
单元的分布，与 nvBench 1.0 在规模、查询长度上的对比，以及三个有代表性示例，
展示一个无歧义种子及其歧义变体。

**流程图。** 五阶段从左到右的流程，每个箭头标注数据量（7,247 个种子产出约 36,000
个变体，每种严重程度 1.25 个变体乘以 4 种类型，第 4 阶段筛选后约保留 30,000 个
验证后变体），质量门控以菱形绘制在第 2、3、4 阶段末尾，并有一个示例贯穿每个阶段。
顶部标注范式标签"可控注入"。

StatQA（逆向合成）和 VisJudge-Bench（自适应生成加专家标注）遵循相同的规格流程，
各自使用范式特定的阶段。当用户的流水线描述含糊或规格不足时，技能以它们为参考模板。

## 用插件实操

按照 [顶层 README](../../../../README.md) 安装插件后，自然语言即可触发。技能会在
"benchmark design is finalized and you need to plan how to actually build the
data"、"source selection, generation strategy, annotation protocol, quality
control"、"How is the data built?" 等短语上自动触发。提供前一阶段的设计文档，
技能会引导完成范式选择、阶段规格、质量控制策略设计和流程图草图。
