# Intro Drafter

## 为什么重要

Introduction 是审稿人读到的头一页半，是否继续读下去的决定就在这里做出。如果六段式的逻辑链断了，即使后面的章节很强，审稿人也会扣分。Introduction 也是论文中最难修复的部分：一个薄弱的方法章节，往往可以通过重写内容来补救；而薄弱的 Introduction，需要的是重新梳理整个故事。

大多数博士生 Introduction 的失败，归因于以下三类之一：段落缺失或顺序错误，局限性与挑战之间没有逻辑衔接，或者贡献点与方法模块没有一一对应。这些都是结构层面的问题，而不是文字层面的问题，也就意味着靠润色修不好，只能靠重建骨架。

intro-drafter 技能接收少量结构化输入（研究领域、局限性、硬约束、核心洞见、关键挑战、方案总览），生成一份六段式大纲，每段都有明确目的、写作要点，以及标注严重性的缺口。它强制执行将 Nice Story 与杂乱罗列的主张区分开来的对齐规则。输出的是大纲，不是段落。学生对照大纲来写段落。

## 核心框架

### 六段式 Flowchart

Introduction 是整篇论文的"压缩版"：研究对象是什么，问题为什么重要，现有工作为何不够，这篇论文贡献了什么，贡献如何映射到章节编号，全部压缩在一到两页之内。六段话按严格顺序承载这一论证。

![Introduction Flowchart](/assets/images/intro_flowchart.png)

图：六段式 Introduction Flowchart：研究背景与运行示例、现有局限、问题本质与目标、关键挑战、方案总览、贡献点。

六段话及其核心目的：

| 编号 | 段落 | 核心目的 |
|---|---|---|
| 1 | 研究背景与动机 | 通过一个具体的运行示例引出研究对象；将论文落地在真实应用场景中 |
| 2 | 现有工作的局限性 | 不超过三条，每条的格式是"已有工作 X 无法处理 Y" |
| 3 | 问题本质与我们的目标 | 明确硬约束（规模、动态性、异构性、端到端开销），推导出目标或问题定义 |
| 4 | 关键挑战 | 不超过三条，每条解释为什么朴素地扩展已有工作行不通 |
| 5 | 方案总览 | 每个模块对应一个挑战，与第四段一一对应 |
| 6 | 贡献点 | 三到四条有编号的要点，每条对应一个章节引用 |

每条箭头都必须成立。任何一条箭头断了，Introduction 读起来就像一组互不相关的主张，而不是一个完整的论证。一位批量阅读五十篇投稿的审稿人，在第一遍就能发现断点。

### 论文类型定位

起草之前，技能首先将论文定位到两条叙事轴之一：

- **技术类论文**（新方法解决已有问题）。叙事核心是 Key Idea 或机制。Goal 只需一句过渡。第三段是轻量的桥接，而不是承重的段落。Alpha-SQL（ICML 2025）是典型示例，见 [case-studies/alpha-sql](../../case-studies/alpha-sql.md)。
- **新问题或新设定类论文**（新问题定义）。Our Goal 或问题定义本身就是贡献。Key Idea 是"为什么这个定义合理且可行"的支撑。第三段是承重的。LEAD（VLDB 2026）是典型示例，见 [case-studies/lead](../../case-studies/lead.md)。

定位影响大纲中重心的落点，以及贡献点的表述方式。技术类论文的贡献强调机制及其证据；新问题类论文的贡献强调问题定义及其意义。

一篇论文若混用两种定位（第三段承重，但贡献按技术类论文写；或者反过来），读起来就会含混。定位决策应该在大纲阶段做一次，然后贯穿每一段。

### 运行示例的纪律

运行示例是论文的锚点。它在第一段以一个具体的失败案例或对比呈现，在方法论中以详细演示重现，在实验中以案例研究重现。只看图表的审稿人，依然能理解论文的故事。

设计原则：真实且具体（来自真实数据或真实场景，不是编造的），简洁但完整（有足够的复杂度来说明问题的难处，但不会让读者淹没），贯穿全文。"LLM 在复杂查询上会产生幻觉"这种抽象表述是弱的；一个具体的失败案例，比如一个自然语言查询导致缺失关键 JOIN 从而产生数千条重复记录的 SQL，是强的。

运行示例不是装饰，它是承重的叙事器件。在第一段之后就抛弃，会让论文的叙事主线断裂，方法论悬在空中与动机脱节。

### 贡献点对齐

每条贡献要点必须满足三项核查：

1. 对应第四段的某个挑战、第五段的某个模块，或某个具体的实验结果。
2. 具体，而不是空洞（单独写"全面的实验验证"不算贡献）。
3. 引用了兑现它的章节编号。

失败模式很容易识别："我们提出了一个新颖的框架"对任何一篇同领域论文都适用；"我们展示了大量实验"没有落地到结果；第四条贡献与任何内容都对不上，是凑数。审稿人见过这些模式几千次了。

### Flowchart 一致性

五条跨段落的逻辑链必须成立：

- 第一段的运行示例，在第五段或某个案例研究预告中被引用。
- 第二段的局限性，推动了第四段的挑战。
- 第三段的目标，与第六段的贡献一对齐。
- 第四段的挑战，与第五段的模块一一对应。
- 第五段的模块，出现在第六段的贡献二或三中。

任何一条链断了，都是 CRITICAL 缺口。技能将这些执行为检查性完整性核查，意味着可以从大纲本身来核验，不需要用户进行任何自证。

五条链合在一起，构成了 Introduction 的叙事主线：沿着链从第一段读到第六段的读者，会得到一个完整的、自洽的论证，说明这篇论文为何存在、交付了什么。发现任何链断裂的读者，则会得到一组松散关联的主张，论文的 Nice Story 分数会急剧下降。

## 核心原则

- **六段式 Flowchart**：研究背景与运行示例、局限性、目标、挑战、方案总览、贡献点，始终按此顺序。
- **局限性不超过三条，挑战不超过三条**：超出这个数量，Introduction 会显得杂乱而不聚焦。
- **技术类论文 vs 新问题/设定类论文**：定位决定第三段的重量。
- **运行示例贯穿全文**：在第一段引入，在方法和实验中复现；不要在第一节之后就抛弃。
- **贡献点对应章节**：每条贡献引用交付它的章节；空洞的贡献会被拒。
- **挑战与模块一一对应**：每个方法模块恰好对应一个挑战。
- **严重性分类**：CRITICAL（链断裂）、MAJOR（审稿人在第一轮会提出）、MINOR（打磨层面）。

## 示例

以 Alpha-SQL（ICML 2025）为例，其 Introduction 在 [case-studies/alpha-sql](../../case-studies/alpha-sql.md) 中有详细分析。技能对这篇论文生成的大纲，直接源自那次分析。

**第一段（研究背景）。** Text-to-SQL 将自然语言查询转化为 SQL，让普通用户和专业用户都能从数据库中获取洞见。LLM 的爆发加速了该领域的发展。运行示例是 BIRD 基准上的一个 Text-to-SQL 查询，其中基线 LLM 生成了一条缺失关键 JOIN 条件的 SQL。

**第二段（局限性）。** 两条局限性，而非三条。训练类方法（在任务特定数据上预训练或微调）需要昂贵的标注数据，且每次出现新的基础模型都要重复这个过程。零样本方法避免了训练，但难以在没有任务特定监督的情况下将 LLM 的通用知识迁移到 SQL 生成上。

**第三段（问题本质与目标）。** 轻量桥接段，因为这是技术类论文。零样本 Text-to-SQL 的本质挑战，是在不训练的情况下将 LLM 知识迁移到特定结构化任务上。隐含目标：用现成的开源 LLM 实现具有竞争力的零样本性能。

**第四段（关键挑战）。** Alpha-SQL 明确跳过了这一段，选择在方法论内部 justify 各技术点的非平凡性。在 AI 顶会页数紧张的情况下，这是可以接受的；但在 SIGMOD 或 VLDB 上，一个单独的挑战段落是被期待的。

**第五段（方案总览）和第六段（贡献点）。** 合并。总览介绍 MCTS 作为搜索骨干，然后呈现两个技术点（LLM-as-Action-Model、自监督奖励函数），每个都附有非平凡性的论证。贡献通过措辞来传达（"我们引入了以下新技术"），而不是单独列一个编号清单。

技能会标记这份大纲的一个缺口：第四段缺失，所以挑战与模块的一一对应关系读者无法核验。在 ICML 这样页数紧张的 AI 顶会，这是可以接受的；在数据库顶会上则会是 CRITICAL 问题。

## 用插件实操

按照 [顶层 README](../../../../README.md#快速安装) 安装插件后，自然语言即可触发。技能会在 "draft the Introduction"、"outline the Introduction"、"intro logic needs clarifying"、"help structure the paper story" 等短语上自动触发。提供技能所需的输入（研究领域、局限性、硬约束、核心洞见、挑战、方案总览），技能返回大纲和 Flowchart 一致性报告。

自然的上游技能是 [tech-paper-template](../tech-paper-template/overview.md)：它生成完整的逻辑骨架（思考模板表格加自洽性核查），intro-drafter 将这个骨架转化为六段式大纲。自然的下游技能是 [figure-designer](../figure-designer/overview.md)：大纲锁定之后，动机示例图应该展示来自第一段的运行示例。
