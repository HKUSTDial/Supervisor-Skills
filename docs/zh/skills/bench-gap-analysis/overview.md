# Benchmark 评估盲区分析

## 为什么重要

NeurIPS Datasets and Benchmarks、ICLR、ICML、ACL、EMNLP 或 CVPR 对 Benchmark 论文
最常见的拒稿模式，是审稿人搞不清楚这个 Benchmark 为何需要存在。审稿人翻开引言，
看到一段话解释作者"为 X 构建了一个新的 Benchmark，因为当前评估存在局限"，
但局限的描述含糊其辞，或者只用数量指标来表达（更多样本、更广的覆盖范围）。
审稿人见过几十篇这样的论文，以贡献仅是渐进式改进（更大的数据集，而非新的评估范式）为由拒稿。
一个可辩护的 Gap Statement，具名、具体、令人意外，是区分被录用和被拒稿的 Benchmark 论文的关键。

Gap Statement 不是自由发挥的创意写作。它遵循一套模板，其背后的推理过程遵循三种已验证
模式之一。这个技能将资深导师默默进行的模式匹配外化出来，让没有直接导师指导的博士生
在投入构建流程之前自行运行这套检查。在设计之前先做评估盲区分析，不是可选项：
论文后续的一切（设计目标、分类学、实验、发现）都依赖于评估盲区。
盲区薄弱，无论流程建得多好，论文都会薄弱。

## 核心框架

这个技能分五步运行，设有强制门禁，阻止在用户确认清晰的 Gap Statement 和研究问题之前
向 Benchmark 设计阶段推进。

### 第一步：调研评估现状

第一步不是写作，而是整理归纳。用户构建一张覆盖目标领域 5 个以上最相关现有 Benchmark
的对比表。这张表会直接成为最终论文的 Table 1，因此这项工作具有直接的后续价值。

| 维度 | Benchmark A | Benchmark B | Benchmark C | ... | 盲区 |
|---|---|---|---|---|---|
| 任务范围 | ... | ... | ... | | |
| 数据规模 | ... | ... | ... | | |
| 评估维度 | ... | ... | ... | | |
| 细粒度级别 | ... | ... | ... | | |
| 真实场景对齐 | ... | ... | ... | | |
| 核心区分项 | 缺失 | 缺失 | 缺失 | | **我们的** |

这一阶段的引导问题：

- 该领域已有哪些主流 Benchmark，每个评估什么？
- 它们共享什么隐含假设？这个假设在真实场景中是否成立？
- 如果一个模型在所有现有 Benchmark 上都得了满分，它真的掌握了底层能力吗？

最后一个问题最为关键。如果答案是"是"，就没有值得发论文的盲区。如果答案是"否"，
那个缺失的要素就是 Gap Statement 的起点。

### 第二步：用三种盲区模式之一识别评估盲区

真实的盲区是现有评估的结构性局限，而非对更多数据或更大数据集的呼吁。这个技能使用
三种已验证的盲区模式，每种附一篇参考论文作为示例。

**Pattern A：维度缺失（Dimension Blindness）。** 现有 Benchmark 测量能力 X，但完全忽略
了相关能力 Y。在现有 Benchmark 上得分高的模型，可能掌握了 X 却未掌握 Y，而没有人
发现这一点，因为根本没有人在测量 Y。StatQA 在数学推理领域发现了这个模式：现有数学
Benchmark 只评分模型是否计算出了正确答案，却忽略了模型是否选择了适当的统计方法。
一个使用了错误的检验方法、但计算结果正确的模型可以得满分，然而其推理根本上是有缺陷的。

**Pattern B：假设违背（Assumption Violation）。** 现有 Benchmark 共享一个在真实场景中
不成立的隐含假设。得分高的模型是在优化这个简化假设，而非真实任务。nvBench 2.0 在
Text-to-Visualisation 领域发现了这个模式：现有 Benchmark 假设每个自然语言查询恰好
对应一个正确的可视化结果。而实际上，真实查询天然具有歧义性。一个类似"展示销售趋势"
的查询，根据上下文，可以合理地产生折线图、柱状图或面积图。现有 Benchmark 会惩罚
合理的备选解释，这恰恰与现实背道而驰。

**Pattern C：粒度不足（Granularity Mismatch）。** 现有评估太粗，无法诊断具体的失败模式。
Benchmark 只产出一个汇总分数，但分数掩盖了模型究竟缺少哪个子能力。VisJudge-Bench
在可视化评估领域发现了这个模式：现有评估孤立地检查表面美学或信息准确性，但真正的
质量需要信达雅三个维度的协同。一张图表可以很漂亮但具有误导性，也可以准确但难以理解。
粗略的"好图/差图"二元评分无法区分这些失败模式。

这个技能要求用户指出观察到的评估盲区最接近哪种模式，并援引一个现有评估漏掉了真实
问题的具体失败案例。如果用户说不出模式名称或具体案例，就还没有准备好写 Benchmark 论文。

### 第三步：用四重验证核查盲区的研究价值

一个盲区只有通过全部四项验证才值得发论文。任何一项失败都意味着盲区需要细化。

- **实际影响**：真实用户或应用会遭遇这个问题。不是理论上的盲区，而是缺失会造成实际危害的盲区。
- **可量化**：盲区可以用具体指标量化，而不只是定性描述。如果盲区是"模型缺乏常识"，
  这项验证失败，因为没有明显的量化指标。
- **非平凡**：现有 Benchmark 不能通过简单扩展来覆盖这个盲区（添加几个样本不算）。
  如果盲区可以通过简单延伸来填补，它就不是结构性的。
- **可操作**：识别这个盲区为模型改进开辟了具体的研究方向。如果盲区有趣但没有人能
  据此行动，论文就不会产生后续工作。

四重验证是短路逻辑。任何一项失败，技能会帮助用户迭代，再继续前进。带着一个未通过
验证的盲区向前推进，会浪费数月时间。

### 第四步：表述 Gap Statement

Gap Statement 是全文最重要的一句话。它遵循结构化模板：

> 针对 [任务] 的现有 Benchmark 专注于 [他们测量什么]，基于 [隐含假设] 这一假设运作。
> 然而，在真实场景中，[假设为何不成立]。这造成了一个关键的评估盲区：[无法被评估的
> 具体能力]，意味着 [后果，即那些表面表现良好的模型会出错的地方]。

该陈述必须具体（命名精确的盲区，而非模糊的局限性）、令人意外（挑战学界习以为常的
某个假设）、后果重大（展示盲区带来的真实危害，而不仅仅是理论上的不完整）。

### 第五步：推导两到三个研究问题

盲区直接推导出 2-3 个研究问题，这些问题将驱动实验章节。每个 RQ 对应论文实验章节的一个子节。

| 覆盖范围 | RQ 模板 | 用途 |
|---|---|---|
| Benchmark 构建 | 如何设计一个 Benchmark 来系统性评估 [评估盲区]？ | 为设计选择提供理由 |
| 模型能力边界 | 现有模型在 [评估盲区] 上失败的程度如何，哪些子能力区分了强模型与弱模型？ | 建立严重程度认知并实现诊断 |
| 人机差距 | 模型与人类专家在 [该维度] 上的差距如何，哪些因素（规模、提示、领域知识）影响了差距？ | 基准真值加可操作洞察 |

## 核心原则

- **三种盲区模式**：维度缺失、假设违背、粒度不足。每个可发表的 Benchmark 盲区都
  符合其中一种。不符合任何一种的，通常只是换皮的数据集发布。
- **四重验证**：实际影响、可量化、非平凡、可操作。四项均须通过。
- **Gap Statement 模板**：结构化五子句形式。具体、令人意外、后果重大。
- **RQ 三领域覆盖**：构建方法、模型能力边界、人机对比。缺少任何一个，论文通常
  会显得洞察单薄。
- **进入设计前的强制门禁**：用户须能在 2-3 句话内阐明盲区，并确认通过全部四项
  验证，才能继续推进至 Benchmark 设计。

## 示例

假设一个博士生提出要为商业智能（BI）查询上的文本转 SQL 系统构建一个 Benchmark。
初始框架是"当前 Text-to-SQL Benchmark 查询复杂度有限，所以我们构建一个更难的"。
这个框架在技能面前立刻失败：不匹配任何盲区模式、未通过非平凡验证（现有 BIRD
Benchmark 可以通过增加更复杂的查询来扩展），也没有清晰的 RQ 覆盖。

技能推动学生调研现有 Benchmark：Spider、BIRD、Spider 2.0、AmbigSQL、KaggleDBQA。
在跨维度梳理过程中，学生发现每个 Benchmark 都在评估单轮查询，且对应唯一正确的 SQL
输出。而真实的商业智能工作涉及多轮对话，分析师通过迭代精炼查询，同一个问题根据
模式解释不同可能产生多个有效 SQL 输出。

这是一个 Pattern B 盲区（假设违背）：所有现有 Benchmark 都共享"Text-to-SQL 是单轮、
无歧义"这一隐含假设。学生通过四重验证：真实 BI 分析师采用多轮工作方式（实际影响），
对话质量可量化（可量化），现有 Benchmark 无法简单扩展，因为它们缺少对话结构
（非平凡），而且这个盲区指向对话式 Text-to-SQL 的新训练范式（可操作）。

Gap Statement 形成："针对商业智能的现有 Text-to-SQL Benchmark 专注于单轮准确性，
基于分析师查询第一次就是完整且明确的假设运作。然而，在真实 BI 工作流中，分析师
通过多轮对话、以模糊的模式引用方式迭代精炼查询。这造成了一个关键的评估盲区：
跨轮次维护模式锚定、追踪引用解析、更新查询语义的能力，意味着一个在单轮 Benchmark
上得满分的模型，在真实 BI 仪表盘场景中仍可能灾难性地失败。"

研究问题：RQ1（如何构建多轮歧义 Text-to-SQL Benchmark）、RQ2（模式追踪、引用解析、
语义更新等哪些子能力区分了强模型与弱模型）、RQ3（模型与人类分析师的差距如何，
领域知识是否有助于弥合差距）。学生现在可以携带锁定的盲区推进至
[bench-design](../bench-design/overview.md)。

StatQA、nvBench 2.0 和 VisJudge-Bench 分别代表一种盲区模式，技能以它们为模板
辅助学生开展自己的工作。StatQA（维度缺失）将数学推理从"计算正确性"拓展到
"方法选择加计算正确性"。nvBench 2.0（假设违背）将 Text-to-Visualisation 从
"一个正确图表"拓展到"具有偏好的多个有效解释"。VisJudge-Bench（粒度不足）将
可视化评估从"孤立的准确性或美学"拓展到"信达雅三者协同的综合体系"。

## 用插件实操

按照 [顶层 README](../../../../README.md) 安装插件后，自然语言即可触发。技能会在
"starting a new benchmark project"、"analysing what existing benchmarks miss"、
"I want to build a benchmark for X" 等短语上自动触发。用一段话描述目标领域，
涵盖已有哪些 Benchmark、你认为缺失什么、以及为什么重要。技能会将提出的盲区映射到
三种模式之一，运行四重验证，并帮助起草 Gap Statement 和研究问题。
