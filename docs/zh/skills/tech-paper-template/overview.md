# Tech Paper Template

## 为什么重要

在起草任何段落之前，技术类论文需要一个完整的逻辑骨架：研究背景、已有工作的具体局限、核心洞见或研究目标、阻止朴素方案成立的技术挑战、应对每个挑战的方法模块，以及论文将主张的贡献点。跳过这一步的学生，最终会遇到方法模块没有解决他们所列局限的情况，或者过度承诺了论文实际无法交付的结果。

这两类失败都代价高昂。方法与局限的错配，会迫使作者在写作中途重写其中一方，而这通常发生在错配变得明显的很晚阶段。过度承诺的贡献更糟：它往往能在写作阶段存活下来，因为作者相信自己的主张；而它会在评审中死去，因为审稿人会指出实验不支持这个贡献。

tech-paper-template 技能通过一张标准化的思考模板表格来填写骨架，将论文定位为技术类或新问题/设定类，并对逻辑链运行四条自洽性核查。输出适合用于导师与学生的头脑风暴、每周进展会议，以及写作开始前的最终规划步骤。

## 核心框架

### 思考模板表格

模板有七个单元格类别，按顺序填写。每个单元格都有具体的内容契约，区分强填写和弱填写。模板是规划阶段论文逻辑的唯一真实来源。

| 阶段 | 填写内容 |
|---|---|
| 研究背景 | 场景、重要性、研究动机。一个紧凑的段落，不是一整页泛泛的热词 |
| 局限性 1 | 某篇具体已有方法中的特定能力缺口 |
| 局限性 2 | 与局限性 1 不同；针对不同维度（成本、鲁棒性、数据） |
| 局限性 3（可选） | 如有额外缺口可写；写两条是可接受的，超过三条则不可 |
| Key Idea 或 Our Goal | 一句话，审稿人可以直接引用。技术类论文：Key Idea。新问题类论文：Our Goal |
| 挑战 1 | 解释为什么 Key Idea 或 Goal 的朴素实现方式行不通 |
| 挑战 2 | 不同的挑战，不同的维度 |
| 挑战 3（可选） | 如果问题确实是三维的，才写第三个挑战 |
| 方法论主题句 | 一句话，引出模块列表 |
| 模块 A（应对挑战 1） | 命名的机制，附一句摘要 |
| 模块 B（应对挑战 2） | 命名的机制，附一句摘要 |
| 模块 C（应对挑战 3） | 命名的机制，附一句摘要 |
| 贡献 1 | 问题定义或框架（第 X 节） |
| 贡献 2 | 主要技术点（第 Y 节） |
| 贡献 3 | 次要技术点或实验评估（第 Z 节） |

只存在于作者脑中的单元格没有被填写；技能会标记其严重性。在每周进展会议中复审模板的浅层，追踪哪些单元格随项目推进而成熟。项目早期，很多单元格会是空的或试探性的；到写作阶段，每个单元格都应该是具体的。

### 论文类型定位

模板根据论文类型有两种不同的使用方式：

- **技术类论文。** 叙事轴是 Key Idea 或机制。Our Goal 只需一句话过渡。贡献强调方法及其证据。Alpha-SQL 和 AFlow 都属于这种类型。
- **新问题或新设定类论文。** 叙事轴是 Our Goal 或问题定义，它本身就是一个贡献。Key Idea 是"为什么这个定义合理且可行"的支撑。贡献强调定义及其意义。LEAD 属于这种类型。

定位影响各单元格的权重。在技术类论文中，Our Goal 是一段简短过渡；在新问题类论文中，Our Goal 是承重的，Key Idea 变成了 Our Goal 的论证。一篇跨越两种定位的论文读起来会含混：审稿人会问，主要贡献究竟是方法还是问题，作者会没有清晰的答案。

### 四条自洽性核查

模板填完之后，四条核查验证逻辑链。每处失败都是 CRITICAL：逻辑链不通，论文就不能被连贯地写出来。

**核查 1：局限性到 Key Idea。** Key Idea 或 Goal 是否解决了所列的局限性？如果 Key Idea 解决的问题从未在局限性中提及，那么局限性写错了，或者 Key Idea 不对，或者两者都有问题。常见失败：局限性描述的是效率缺口，但 Key Idea 是关于准确率的。这样的论文无法声称 Key Idea 解决了局限性。

**核查 2：Key Idea 到挑战。** 挑战是否自然地从实现 Key Idea 的过程中产生？如果挑战看起来是编造的，那很可能真的是。常见失败：Key Idea 是"用 MCTS 做 Text-to-SQL"，但所列挑战是"如何解析 SQL"和"如何访问数据库"，这些是与 MCTS 选择无关的通用问题。真正的 MCTS 挑战（搜索空间设计、奖励函数、rollout 预算）反而缺席。

**核查 3：挑战到方法论。** 每个方法模块是否应对了一个挑战？一一对应是目标。没有对应挑战的模块，要么缺乏论证，要么在解决一个与论文主张不同的问题。没有对应模块的挑战，是一个未解决的缺口。多对一或一对多的映射都是警告信号。

**核查 4：方法论到贡献。** 贡献点是否覆盖了每个模块或实验结果？泛泛到适用于任何论文的贡献（"新颖的框架"、"大量实验"）无法通过这项核查。承诺了实验不支持的结果的贡献，也无法通过。每条贡献都应当指向一个具体的方法或实验交付。

### 作为副产品的方法论大纲

所有核查通过之后，模板生成方法论大纲：主题句、每个模块的小节名称、每个模块的一句摘要。这成为论文方法章节的骨架。在起草段落之前运行这一步可以节省后续的返工：跳过大纲阶段的作者往往在写到第三节中途才发现，他们的模块其实与挑战不匹配。

## 关键核查项

- **论文类型定位**与实际贡献一致（技术类论文没有被强行塞进新问题框架，反之亦然）。
- **局限性具体且可引用**，不是泛泛的套话。
- **Key Idea 或 Goal 是一句话**，审稿人可以直接引用。
- **挑战从 Key Idea 的实现中产生**，不是为了为模块提供论证而编造的。
- **方法模块与挑战一一对应。**
- **贡献点对应方法模块和具体章节**，每条都有第 X 节的指针。
- **四条自洽性核查全部通过**，否则将链断裂标记为 CRITICAL。

## 示例

以 Alpha-SQL（ICML 2025）填入模板为例。完整分析见 [case-studies/alpha-sql](../../case-studies/alpha-sql.md)；这里只给出模板层面的摘要。

| 阶段 | Alpha-SQL 内容 |
|---|---|
| 研究背景 | Text-to-SQL 使数据库访问民主化；LLM 推动了快速进展 |
| 局限性 1 | 训练类方法需要大量标注数据，且每次基础模型更换都要重新训练 |
| 局限性 2 | 零样本方法在不用监督的情况下，难以将通用 LLM 知识迁移到结构化 SQL 任务 |
| Key Idea | 用 MCTS 动态生成并探索 SQL 构造动作，以 LLM 推理为引导 |
| 挑战 | （在 AI 顶会页数限制下，折叠进了方法论） |
| 方法论主题句 | Alpha-SQL 利用带两项新技术的 MCTS 框架 |
| 模块 A | LLM-as-Action-Model：在每步动作后调用 LLM 生成链式推理，存储在每个节点 |
| 模块 B | 自监督奖励函数：跨每条推理路径的多个采样 SQL 查询计算自洽性分数 |
| 贡献 1 | 无需微调的即插即用 Text-to-SQL 框架 |
| 贡献 2 | MCTS 中的 LLM-as-Action-Model（第 3 节） |
| 贡献 3 | 基于执行自洽性的自监督奖励函数（第 3 节） |
| 贡献 4 | 使用开源 LLM 在 BIRD 开发集上达到 69.7% 准确率，显著超过现有零样本基线（第 4 节） |

运行四条核查：

- 核查 1（局限性到 Key Idea）：局限性 1 是训练成本；Key Idea 完全避免了训练。局限性 2 是零样本知识迁移；Key Idea 用 MCTS 加链式推理来引导迁移。两条都通过。
- 核查 2（Key Idea 到挑战）：形式上未解决，因为论文没有列举挑战。隐性挑战（没有真实 SQL 的 MCTS 奖励设计、SQL 构造的动作空间设计、跨搜索状态的推理连贯性）都在方法论章节内处理了。
- 核查 3（挑战到方法论）：模块 A 和 B 分别对应推理连贯性和奖励设计的隐性挑战。审稿人会更希望看到一个显式的挑战段落，但方法论确实覆盖了这两点。
- 核查 4（方法论到贡献）：贡献 1-3 对应框架和模块；贡献 4 报告了结果。全部通过。

模板练习揭示了 Alpha-SQL 为了应对页数限制而做出的权衡：将挑战折叠进了方法论。对于数据库顶会（SIGMOD、VLDB）的投稿，学生在提交前应该把挑战段落还原。

## 用插件实操

按照 [顶层 README](../../../../README.md#快速安装) 安装插件后，自然语言即可触发。技能会在 "paper skeleton"、"paper logic chain"、"thinking template"、"paper-structure planning" 等短语上自动触发。典型用法：先运行 [idea-evaluator](../idea-evaluator/overview.md) 确认 idea 值得追求，再运行 tech-paper-template 锁定骨架，然后运行 [intro-drafter](../intro-drafter/overview.md) 将骨架转化为 Introduction 大纲。
