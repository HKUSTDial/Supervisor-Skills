# Benchmark 实验与洞察设计

## 为什么重要

Benchmark 论文的实验章节与技术论文的实验章节有着根本不同的目标。技术论文证明
一个方法超越了基线；Benchmark 论文揭示模型在哪里、为什么失败，以及这对未来研究
意味着什么。用技术论文的思维方式写 Benchmark 实验章节，会产生 NeurIPS Datasets
and Benchmarks、ICLR、ICML 和 CVPR 上第二常见的拒稿模式：一张巨大的排行榜表格，
却没有提炼出任何洞察。审稿人看了排行榜，点头承认某个前沿模型平均超过了另一个，
然后问："然后呢？"没有 "Finding X" 提炼模式的 Benchmark 论文，没有可供未来论文
引用的发现。

这个技能围绕研究问题和 Finding X 模式来设计实验章节，后者是 Benchmark 论文的
标志性写作技法。它覆盖了四个维度的基线覆盖、评估协议（含 LLM-as-Judge 验证和
人类基准线）、RQ 驱动的分析结构、整体性能表、Finding 提炼模式、案例研究选择、
可选的配套方法实验，以及研究机会梳理。强制门禁强制执行思维转变：不要设计实验
来证明优越性；要设计实验来诊断模型行为，生成可操作的洞察。

## 核心框架

技能按顺序运行八个部分，每个部分对应论文实验材料的一个章节。

### 1. 四个维度的基线模型选择

审稿人会检查基线集是否具有代表性。覆盖多个维度以实现全面对比；最少 10-15 个模型。

| 维度 | 类别 | 目的 |
|---|---|---|
| 开源 vs. 闭源 | 闭源前沿模型 vs. LLaMA、Qwen、Mistral、DeepSeek | 专有模型与可访问模型之间的能力差距 |
| 模型规模 | 7B -> 13B -> 70B -> 100B+ | 能力随参数规模的变化 |
| 架构 | Decoder-only vs. Encoder-Decoder；纯文本 vs. 多模态 | 架构特定的优势和弱点 |
| 专业化 | 通用 vs. 领域专用（代码、数学、视觉） | 专业化训练是否能迁移 |

覆盖全部四个维度的基线集，预先化解了"为什么没有测试 X？"的审稿人质疑，
因为每个主要方向都有代表。

### 2. 评估协议

**2.1 评估设置。** 定义模型如何与 Benchmark 交互：

| 要素 | 关键问题 | 指引 |
|---|---|---|
| 输入格式 | 模型如何接收输入？提供什么上下文？ | 指定提示模板、输入结构、特殊格式要求 |
| 输出提取 | 如何解析和评分模型输出？ | 定义提取规则、解析逻辑、边界情况处理 |
| 提示策略 | 零样本、少样本（1/3/5）、CoT、领域知识提示、工具调用 | 测试多种；报告哪种有帮助或有害 |
| 评估方式 | 自动指标、LLM-as-Judge、人工评估 | 至少使用两种；将 LLM 评判与人工对齐验证 |
| 指标选择 | 选择什么指标，为什么？ | 说明指标选择理由；推荐一个主要指标，细粒度指标作为辅助 |
| 重复次数 | 每个模型 1-5 次 | 对非确定性设置报告均值加减标准差 |
| 温度 | 0 用于可复现性，大于 0 用于多样性分析 | 记录选择 |

**2.2 人类基准实验。** 如果包含人工评估，说明参与者构成（数量、背景、招募标准）、
实验协议（任务说明、时间限制、标注界面）、标注者间可靠性（一致性指标加阈值），
以及对比设计（人类和模型评估相同样本）。

**2.3 基线公平性。** 为每个基线均等地记录优化努力。审稿人越来越意识到"基线弱化"
做法，即故意对竞争对手的超参数优化不足。每个基线都应使用已知最佳配置。

### 3. RQ 驱动的实验结构

将每个实验围绕一个研究问题来组织。每个 RQ 驱动论文中的一个分析子节：

```
RQ[N]：[问题]
 ├── 假设：[预期发现]
 ├── 实验：[具体比较或分析]
 ├── 变量：[变化的 vs. 控制的]
 ├── 指标：[测量什么]
 ├── 可视化：[图表类型]
 └── 预期发现：[该分析能产出什么洞察]
```

常见分析类型表，附优先级和可视化：

| 分析 | 揭示内容 | 可视化 | 优先级 |
|---|---|---|---|
| 整体性能 | 模型能力全景 | 大表：所有模型 × 所有指标 | 必须 |
| 类别细分 | 各分类学类别上的性能 | 分组柱状图或热力图 | 必须 |
| 难度梯度 | 随难度增加性能如何下降 | 折线图（x=难度，y=分数） | 必须 |
| 错误分类 | 模型犯什么类型的错误 | 堆叠柱状图、饼图加错误示例 | 高 |
| 模型行为偏差 | 模型是否有系统性倾向（分数膨胀、过于保守、偏好特定答案类型） | 分布密度曲线、校准图 | 高 |
| 人机对比 | AI 在哪里匹配、超越或落后于人类 | 雷达图或配对比较柱状图 | 高 |
| 提示影响 | 策略如何影响性能 | 消融表（行=模型，列=策略） | 中 |
| 规模效应 | 模型规模如何影响能力 | 折线图（x=参数量，y=分数） | 中 |
| 跨维度相关性 | 哪些能力相互关联 | 相关性热力图 | 可选 |

### 4. 整体性能表

论文中最大、最重要的表（通常是 Table 3）。设计指南：

- 每列最佳结果加粗；次优加下划线。
- 按类别分组（闭源、开源、专用）。
- 如适用，将人类表现作为上界。
- 如有洞察价值，添加平均和最差情况行。

### 5. Finding X 模式（标志性技法）

每次主要分析后，提炼并加粗高亮一个编号发现。格式规范的 Finding 示例：

> **Finding 1.** Few-shot learning and domain-knowledge inclusion help
> LLMs, whereas Chain-of-Thought tends to slightly degrade performance
> in smaller models.

> **Finding 2.** All tested models perform below random baseline on
> [specific sub-category], revealing a fundamental capability gap in
> [specific skill].

> **Finding 3.** Model scale improves [Dimension A] performance
> monotonically, but has no significant effect on [Dimension B],
> suggesting these capabilities have different scaling properties.

每个 Finding 必须：令人意外或不显而易见（不只是"更大的模型更好"）、具体（命名
具体模型、类别或条件）、可操作（暗示未来研究应聚焦的方向），以及有数据支撑
（由其上方的分析直接支持）。

每个 Finding 的**微结构模板**：

```
[附数据的分析段落。]

**Finding N.** [引导句：一句话陈述核心结论。]
[证据：具体数字、模型、条件。]
[限定：范围和边界条件。]
[前向指引：这对未来工作意味着什么。]
```

Finding 模式是让 Benchmark 论文可引用的关键。一篇没有 Finding 的 Benchmark 论文，
不会产生后续工作。

### 6. 案例研究设计

包含 2-4 个案例研究，提供定性深度：

| 类型 | 用途 | 格式 |
|---|---|---|
| 成功案例 | 展示 Benchmark 能区分能力 | 强模型 vs. 弱模型在同一输入上的对比 |
| 失败案例 | 揭示特定模型局限 | 模型输出加标注（在哪里失败、为什么失败） |
| 令人意外的案例 | 突出反直觉行为 | 建立预期，然后展示意外结果 |
| 边界案例 | 测试能力边界 | 微小变化导致正确翻转为错误 |

### 7. 配套方法实验（如适用）

如果在 [bench-design](../bench-design/overview.md) 中提出了配套方法，实验章节
也应报告其评估：与基线在新 Benchmark 上的对比、方法组件的消融研究、分类别
提升分析（方法在哪里帮助最大）、泛化测试（提升是否能迁移到其他 Benchmark），
以及下游应用验证（Benchmark 或方法是否能提升真实场景性能，如 VisJudge-Bench
通过下游可视化质量提升来验证）。

### 8. 研究机会

从 Finding 中推导 3-5 个未来方向。典型方向：如何提升模型在已识别盲区上的能力、
任务的人机协作模式、Benchmark 扩展（新模态、语言、领域、难度级别）、受发现启发
的训练方法改进、对某个特定 Finding 背后原因的理论理解。

## 核心原则

- **四个基线维度**：开源 vs. 闭源、规模、架构、专业化。覆盖全部四个，或预先
  化解审稿人的质疑。
- **RQ 驱动结构**：每个分析子节对应一个研究问题。没有孤立的分析。
- **Finding X 模式**：加粗高亮、编号、令人意外、具体、可操作、有数据支撑。
  Benchmark 论文的标志性写作技法。
- **基线公平性纪律**：为每个基线均等记录优化努力。不做"基线弱化"。
- **LLM-as-Judge 验证规则**：任何 LLM-as-Judge 协议都必须报告与人工标注的相关性。
  没有验证，没有录用。
- **案例研究四类覆盖**：成功、失败、令人意外和边界。共 2-4 个案例。

## 示例

考虑 VisJudge-Bench 完成数据构建后的情况。盲区是粒度不足：现有可视化评估孤立地
检查美学或准确性。分类学是三个顶层维度（信、达、雅），每个含两个子维度。

**基线。** 四个维度的 15 个模型。来自多家供应商的闭源前沿模型。多规模开源模型：
LLaMA-3-8B、LLaMA-3-70B、Qwen2.5-7B、Qwen2.5-72B、DeepSeek-V3、Mistral-Large。
专门用于多模态的：GPT-4V、InternVL、LLaVA-NeXT 加两个其他视觉语言模型。
领域专用的：一个专有可视化模型加两个在图表数据集上训练的视觉语言模型。

**评估协议。** 两种评估方式：信度（Fidelity）的自动指标（生成图表规格与基准图表规格
的 BLEU 风格重叠），以及达度（Expressiveness）和雅度（Aesthetics）的 LLM-as-Judge。
LLM-as-Judge 与人类博士生评分者对齐，每个子维度报告 Pearson 相关系数。附录中记录
了基线公平性：每个模型使用其最佳已知提示策略，跨模型使用相同的 shot 数量。

**RQ 结构。** RQ1（模型在六个子维度各自的表现如何？）驱动整体性能和类别细分分析。
RQ2（性能如何随难度级别变化？）驱动难度梯度分析。RQ3（模型在哪里系统性地失败，
主导的错误类型是什么？）驱动错误分类和案例研究。

**Finding。** Finding 1：闭源模型在雅度上大幅领先于开源模型，但在信度上差距缩小
（任务偏事实性时开源模型追上来了）。Finding 2：没有任何模型在达度子维度上超过
60%，揭示了图表层面沟通清晰度推理能力的根本差距。Finding 3：专门针对图表数据
训练的视觉语言模型，在雅度上表现逊于 GPT-4V，尽管在信度上更优，表明专业化以
审美判断换取了事实准确性。

**案例研究。** 四个案例：一个成功案例，GPT-4o 正确识别了一个误导性轴刻度；一个
失败案例，所有测试模型都遗漏了一个色盲可访问性问题；一个令人意外的案例，一个
较小的模型在一个微妙的审美判断上优于较大的模型；一个边界案例，微小的布局变化
导致模型的评分从正面翻转为负面。

**配套方法实验。** VisJudge 模型（在 Benchmark 细粒度评分上 GRPO 训练）与 GPT-4o
在保留数据上对比评估。分子维度提升分析展示了 GRPO 在哪里帮助最大。泛化到
ChartQA 和 MiniVQA 的测试确认了提升能够迁移。下游应用验证表明，在端到端图表生成
流水线中，VisJudge 模型将最终图表质量提升了可量化的百分比。

**研究机会。** 三个方向：扩展到交互式仪表盘（当前 Benchmark 只有静态图表），
扩展到低资源语言的图表标题，将审美-事实权衡作为通用视觉语言模型训练问题来研究。

StatQA 的实验设计遵循相同结构，有不同的 RQ 和 Finding。nvBench 2.0 遵循相同结构，
其可控注入的严重程度梯度驱动了独特的难度梯度分析。

## 用插件实操

按照 [顶层 README](../../../../README.md) 安装插件后，自然语言即可触发。技能会在
"the benchmark data exists or is being built and you need to plan baselines"、
"plan baselines, evaluation protocols, fine-grained analyses"、"What experiments
yield the deepest insights?" 等短语上自动触发。提供来自
[bench-gap-analysis](../bench-gap-analysis/overview.md) 的研究问题和来自
[bench-design](../bench-design/overview.md) 的设计文档，技能会引导完成基线选择、
评估协议、RQ 驱动分析设计和 Finding 提炼规划。
